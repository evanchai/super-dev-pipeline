# Dev Pipeline — 全流程开发引擎

**每次开发任务自动执行。不需要用户触发。**

```
[用户批评/纠正] ──→ 立即触发 evolve ──→ 继续当前阶段
        ↓（全局 hook，覆盖所有阶段）

        ┌──────────── 需求不合理 ←──────────────┐
        ↓                                       |
P1 产品设计 → P2 UI/UX 设计 → P3 实现计划 → P4 编码实现
        ↑         |                ↑              |
        └─ 技术不可行 ─┘            └── code review 🔴 blocker
                                                  ↓
                                   P5 验证检查 → 发现问题 → 回 P4 或 P2
                                        ↓
                                   P6 发布上线 → e2e 失败 → 回 P4（最多 3 次）
                                        ↓              ↑
                                   P7 上线验证    （异步：部署等待期间并行做文档/检查）
                                        ↓
                                   P8 自我进化 → 更新 Pipeline → 下次更强

每个阶段转换时写 PIPELINE_STATE.json → 新 session 自动恢复
```

## Session 恢复：状态持久化

**每次阶段转换时，写入项目根目录 `PIPELINE_STATE.json`：**

```json
{
  "phase": "P4",
  "task": "feat: 用户登录",
  "completedPhases": ["P1", "P2", "P3"],
  "currentStep": "4a-tdd",
  "pendingIssues": ["review blocker #2"],
  "designDoc": "docs/design/2026-03-21-login.md",
  "timestamp": "2026-03-21T15:00:00Z"
}
```

**规则：**

1. **每次阶段转换必须更新** — 进入 P2 前写一次，进入 P3 前写一次，以此类推
2. **新 session 开头先检查** — 如果项目根目录有 `PIPELINE_STATE.json`，读取并恢复：
   - 展示上次进度："上次停在 P4（编码实现），已完成 P1-P3，还有 1 个 review blocker"
   - 问用户："继续上次的任务？还是开始新任务？"
3. **24 小时过期** — timestamp 超过 24h，提示用户：
   - "上次任务已经 48 小时了（P4 编码实现），继续还是放弃？"
   - 用户说放弃 → 删除状态文件，从头开始
4. **P8 完成后清理** — pipeline 全部走完，删除 `PIPELINE_STATE.json`
5. **轻量模式不写状态** — diff < 20 行的快车道不需要持久化
6. **`.gitignore` 必须包含** — `PIPELINE_STATE.json` 不应被 commit

**状态文件字段说明：**

| 字段 | 含义 |
|------|------|
| `phase` | 当前所在阶段（P1-P8） |
| `task` | 任务描述（一句话） |
| `completedPhases` | 已完成的阶段列表 |
| `currentStep` | 阶段内的具体步骤（如 "4a-tdd", "4b-review", "6-e2e-retry-2"） |
| `pendingIssues` | 待解决的问题列表（review blocker、test failure 等） |
| `designDoc` | 设计文档路径（方便恢复时快速找到上下文） |
| `timestamp` | 最后更新时间（ISO 8601） |

## Auto-Loop 模式（批量任务自动循环）

**适用场景：** 用户提前批准了一批明确任务（"修这 5 个 bug"、"给所有项目加 rate limiting"、"跑完这个 TODO 清单"），每个任务的"做什么"已经确定，不需要 P1/P2 设计确认。

### 触发条件

用户明确给出**任务清单**（多个独立任务），且任务描述足够具体（不需要设计讨论）。

### 流程

```
用户给任务清单
  ↓
写入 TASK_QUEUE.json（任务列表 + 状态）
  ↓
┌─→ 取第一个 status:"pending" 的任务
│     ↓
│   判断入口（通常 P3 或 P4）
│     ↓
│   执行 pipeline P3/P4 → P5 → P6 → P7
│     ↓
│   更新 TASK_QUEUE.json（标记 completed + 结果摘要）
│     ↓
│   还有 pending 任务？──→ 是 → 回到顶部
│                          ↓
│                         否 → P8 进化 + 汇报总结
└─────────────────────────────┘

安全阀：
- MAX_TASKS = 10（单次循环不超过 10 个任务）
- 单个任务连续失败 3 次 → 标记 failed，跳到下一个
- 任何任务触发安全问题（密钥泄露等）→ 立即停止整个循环
```

### TASK_QUEUE.json 格式

```json
{
  "mode": "auto-loop",
  "maxTasks": 10,
  "startedAt": "2026-03-22T23:00:00Z",
  "tasks": [
    {
      "id": 1,
      "description": "给 logos 项目加 CORS origin 白名单",
      "project": "logos",
      "status": "completed",
      "result": "已添加白名单，push 完成，线上验证通过",
      "completedAt": "2026-03-22T23:15:00Z"
    },
    {
      "id": 2,
      "description": "修复 museum API 403 错误",
      "project": "museum",
      "status": "in_progress",
      "entryPhase": "P4",
      "retries": 0
    },
    {
      "id": 3,
      "description": "给 tunes 加 error boundary",
      "project": "tunes",
      "status": "pending"
    }
  ]
}
```

### 状态流转

```
pending → in_progress → completed
                     → failed（3 次重试后）
                     → blocked（需要人类决策，跳过）
```

### 规则

1. **不进 P1/P2** — auto-loop 的任务不需要设计讨论，入口是 P3 或 P4
2. **任务间独立** — 一个任务失败不影响其他任务继续
3. **每个任务完成后更新 TASK_QUEUE.json** — 断点恢复时知道哪些做了哪些没做
4. **循环结束统一汇报** — 完成/失败/跳过各多少，每个任务的结果摘要
5. **不自动创建新任务** — 执行过程中发现的新问题标记为 "discovered"，汇报给用户，不自行加入队列
6. **`.gitignore` 必须包含** — `TASK_QUEUE.json` 不应被 commit

### 与 PIPELINE_STATE.json 的关系

- `TASK_QUEUE.json` 管**任务间**的进度（哪些任务做了、哪些没做）
- `PIPELINE_STATE.json` 管**单个任务内**的进度（当前在 P 几、什么步骤）
- auto-loop 模式下两个文件共存，新 session 恢复时先读 `TASK_QUEUE.json` 找到当前任务，再读 `PIPELINE_STATE.json` 找到任务内的进度

### 全局 Hook：用户批评立即 evolve

**在任何阶段**，检测到以下信号时，暂停当前工作，立即运行 `evolve` skill，完成后继续：
- 用户指出速度慢、走弯路、漏步骤
- 用户问"为什么不…"/"你有没有反思"
- 用户说"这应该已经在 pipeline 里"/"你之前犯过这个错"

## 入口选择（自动判断）

| 任务类型 | 规模 | 入口阶段 | 示例 |
|----------|------|---------|------|
| 新项目 | 大 | **P1** 产品设计 | "做一个新的 web app" |
| 新功能 | 大 | **P1** 产品设计 | "加一个分享功能" |
| UI 改动 | 大 | **P2** UI/UX 设计 | "改一下首页布局" |
| 架构变更 | 大 | **P3** 实现计划 | "从 Firebase 迁移到 Supabase" |
| Bug 修复 | 中 | **P4** 编码实现（走调试路径） | "修复登录报错" |
| 单模块逻辑改动 | 中 | **P4** 编码实现 | "优化分享流程" |
| 文案/颜色/间距 | 小 | **P4** 轻量模式 | "改个文案" "调个颜色" |
| 纯性能优化 | 中 | **P5** 验证检查 | "优化加载速度" |

**规则：从入口阶段开始，自动走完后续所有阶段。**

### 改动规模定义与测试分层

<HARD-GATE>
改动规模决定测试深度。不允许自己把中/大改动降级为小改动来跳过测试。
判断标准是**改了什么**，不是**改了多少行**。
</HARD-GATE>

| 规模 | 定义 | P4 测试 | P5 测试 | P6 测试 |
|------|------|---------|---------|---------|
| **小** | 只改文案/颜色/间距/配置/typo，不碰逻辑、不碰 API、不碰数据流 | 无 | build | 无 |
| **中** | 改一个模块的逻辑，或 bug 修复 | 相关模块测试 | **全量单元测试**（回归检测） | **e2e**（如有） |
| **大** | 跨模块、改 API、改数据模型、新功能 | 相关模块测试 | **全量单元测试** | **完整 release-flow** |

**判断依据（按顺序检查）：**
1. 碰了 API 端点 (`api/*.ts`) → 至少**中**
2. 碰了认证/支付/数据模型 → 至少**中**
3. 碰了多个组件或跨模块依赖 → **大**
4. 只碰了一个文件的非逻辑部分 → **小**

**小改动流程（快车道）：**
```
直接改 → npm run build → git diff 检查无密钥 → commit + push → 上线验证
```

**跳过的步骤：** P1-P3 设计/计划、两阶段 Review、全量测试、e2e
**保留的步骤：** 构建验证、安全检查（铁律 5）、上线验证

## 五道铁律（Hard Gates）

贯穿整个 Pipeline 的不可违反原则：

| # | 铁律 | 含义 |
|---|------|------|
| 1 | **设计先行** | 设计未经用户确认前，禁止写任何代码 |
| 2 | **测试先行** | 中/大改动必须先写失败测试再写代码（小改动豁免） |
| 3 | **根因先行** | Bug 修复必须先找到根因，禁止"试试改这个看看" |
| 4 | **验证先行** | 没跑验证命令就不能说"完成了"/"通过了" |
| 5 | **安全先行** | commit 前必查密钥泄露 |
| 6 | **现实先行** | 读 memory 后必须比对实际代码（package.json、imports），memory 过时以代码为准，立即更新 memory |

**违反任何一条铁律 = 流程失败。没有例外、没有"这次简单所以跳过"。**

## 阶段间挑战机制

**核心原则：每个阶段的产出必须经过下游审查，不合格则打回上游修复。**

### 挑战回路一览

| 审查点 | 谁审查 | 审查什么 | 不通过则 |
|--------|--------|---------|---------|
| P1→P2 | P2 UI/UX 审视 P1 产出 | 需求是否可设计？用户场景是否合理？ | 回 P1 补充需求 |
| P2→P3 | P3 架构审视 P2 产出 | 设计是否技术可行？性能是否可接受？ | 回 P2 调整设计 或 P1 调整需求 |
| P3→P4 | P4 开始前自检 | 计划是否过度设计？步骤是否可执行？ | 回 P3 简化 |
| P4→P5 | 两阶段 Review | Spec 合规 → 代码质量 | 回 P4 修复代码 |
| P5→P6 | 验证门禁 | 安全/设计/无障碍/性能全过？ | 回 P4（代码问题）或 P2（设计问题） |
| P6→P7 | `release-flow` e2e | Playwright 测试全过？ | 回 P4 修复 → 重新 P6（最多 3 次） |

### 挑战执行规则

1. **自动执行** — 不问用户"要不要 review"，直接审查
2. **只打回有明确问题的** — 不要为了审查而审查，没问题就过
3. **打回时说清原因** — 明确指出哪里不合格、为什么、建议怎么改
4. **最多回弹 3 次** — 同一个问题来回超过 3 次，说明方案本身有问题，暂停并汇报用户（统一上限：调试 3 次、e2e 重试 3 次、挑战回弹 3 次）
5. **小改动轻审查** — 仅限**小规模**改动（文案/颜色/间距），bug fix 是**中规模**，必须走完整流程

## Phase 1: 产品设计

**Goal：想清楚做什么、为谁做、不做什么。**

<HARD-GATE>
铁律 1：设计先行。
设计未经用户确认前，禁止写任何代码、创建任何文件、调用任何实现 skill。
无论项目看起来多简单，都必须走完设计流程。"简单"项目正是未经审视的假设最浪费时间的地方。
</HARD-GATE>

### 1.1 流程

```
探索上下文 → 逐个提问 → 提出 2-3 方案 → 分段呈现设计 → 用户确认 → 写设计文档 → 进入 P2
```

### 1.2 调研先行（提问之前必须完成）

**在问用户任何问题之前，先完成调研。** 带着调研结果再问，问题质量更高。

1. **GitHub 搜索** — `gh search repos` / `gh search code` 找现有实现、模板、模式
2. **包注册表** — npm / PyPI 搜索是否有现成库解决 80%+ 的问题
3. **项目上下文** — 检查现有文件、文档、最近 commit
4. **竞品分析** — 同类产品怎么做的？有没有可以借鉴的交互模式？

**调研产出：** 带着发现开始提问（"我搜了一下，发现 XX 库可以做 YY，你觉得...？"）

### 1.3 评估规模

- 如果包含多个独立子系统，**先帮用户拆解**再逐个设计
  - 多子系统项目：标记拆解 → 每个子系统走独立的 P1→P6 循环
  - 单系统项目：正常流程

### 1.4 逐个提问

**一次只问一个问题。** 不要一口气列出多个问题让用户回答。

- **优先多选题** — 降低用户回答成本，给出推荐选项并说明推荐理由
- **开放题也可以** — 但每条消息只问一个
- 聚焦理解：目的、约束、成功标准、目标用户
- 如果某个话题需要深入，拆成多个问题分次问
- 问题数量根据项目复杂度弹性调整（简单项目 2-3 个，复杂项目 5-8 个）

**反模式（禁止）：**
- ❌ 一次列出 4+ 个问题
- ❌ 没有推荐选项的开放式提问（"你想怎么做？"）
- ❌ 不了解上下文就开始问细节

### 1.5 提出方案

- 提出 **2-3 个不同方案**，附权衡分析
- **带推荐** — 说清楚推荐哪个、为什么
- 方案之间应有实质差异（不是微调），让用户在方向层面做选择
- 领先推荐方案，然后解释其他选项为什么不推荐

### 1.6 分段呈现设计

理解清楚后，分段呈现设计方案：

- 每段复杂度自适应：简单的几句话，复杂的详细展开
- **每段确认后再继续** — 不要一次性铺完所有内容
- 覆盖：核心功能、用户流程、数据流、错误处理
- 用户不认可则返回修改，不要硬推

**设计隔离原则：**
- 系统拆分为小单元，每个单元有单一职责
- 每个单元通过明确接口通信，可独立理解和测试
- 能回答三个问题：它做什么？怎么用？依赖什么？

### 1.7 成功指标（Definition of Done）

在设计确认前，必须回答一个问题：**"怎么判断这个功能成功了？"**

- 定义 1-3 个可观测的成功标准（不需要量化，但需要可验证）
- 示例：
  - "用户能在 3 步内完成核心流程"
  - "页面加载 < 2s"
  - "错误率不高于部署前"
  - "我自己会日常使用它"
- 写入设计文档的 **Success Criteria** 章节
- P7 上线验证时回来对照检查

### 1.8 YAGNI 原则

- **砍掉所有非必要功能** — 这是最重要的设计决策
- MVP 只留 must-have，nice-to-have 明确标记为后续迭代
- 宁可功能少而精，不要功能多而糙

### 1.9 写设计文档

设计确认后，写入项目目录：
- 路径：`docs/design/YYYY-MM-DD-<topic>.md`
- 内容：问题定义 + 功能列表 + Non-goals + 技术约束 + 成功指标 + 用户确认的方案
- commit 到 git

**调用 skill：**
- `product-manager` → PRD、RICE 优先级、MVP 裁剪
- `renaissance-architecture` → 第一性原理，避免做衍生品
- `human-architect-mindset` → 系统架构决策
- `/office-hours`（gstack）→ 新项目/新产品方向时，用 YC 六问逼出真正需求（可选，与 product-manager 互补）

**产出：** 设计文档（用户已确认）

**质量门禁：** 进入 P2 前自检——
- [ ] 问题定义清晰（一句话说得清）
- [ ] 功能列表有优先级
- [ ] Non-goals 已明确
- [ ] 用户已逐段确认设计方案
- [ ] 设计文档已写入并 commit

## Phase 2: UI/UX 设计

**Goal：确定怎么做、长什么样。**

**触发条件：** 涉及新页面、新组件、交互变更时执行；纯后端/纯逻辑跳过。

1. 确定风格、布局、交互模式
2. 遵循项目设计语言（dark theme, gradient accents, mobile-first）
3. 无障碍：aria-label、keyboard、contrast
4. 排版：引号、破折号、间距、层次

**UI skill（简化为 2 层）：**

| 层 | Skill | 职责 |
|----|-------|------|
| 约束 | `baseline-ui` | 硬约束 + 排版 + 无障碍，自动执行 |
| 创意 | `frontend-design` | 设计+实现，在约束内自由发挥 |

其他 UI skill（`ui-ux-pro-max`, `ckm-ui-styling`, `web-design-guidelines`, `ui-typography`）按需调用，不强制。

**↑ 挑战 P1：** 设计过程中如果发现需求不合理（场景缺失、用户流程说不通），回 P1 补充。

**产出：** 组件设计方案 or 直接输出代码

**质量门禁：** 进入 P3 前自检——
- [ ] 设计符合项目风格（dark theme, mobile-first）
- [ ] 交互流程完整（用户能走完核心流程）
- [ ] 无障碍基础到位（对比度、键盘可达）

## Phase 3: 实现计划

**Goal：把设计转化为可执行的 bite-sized 步骤。**

**触发条件：** 新项目、跨模块改动、数据模型变更时执行；简单改动跳过。

### 3.1 技术选型

1. 技术选型（参考 CLAUDE.md 模板）
2. 目录结构（`src/` 布局推荐）
3. API 设计（Vercel Functions，输入验证，错误处理）
4. 数据模型（Supabase/Firebase schema）

### 3.2 写实现计划

计划必须是 **bite-sized 可执行步骤**，不是模糊的大纲：

**计划文档头部（必填）：**
```markdown
# [Feature Name] 实现计划
**目标：** [一句话]
**架构：** [2-3 句]
```

**每个步骤必须包含：**

**步骤模板：**

```markdown
2. 写最小实现 → `npm run test` → 期望 PASS
3. 验证：`npm run build` → exit 0
```

**计划原则：**
- DRY、YAGNI、TDD、频繁 commit

- 复杂项目拆成多个独立的 task，可并行分发给 subagent

**↑ 挑战 P2：** 如果设计方案技术不可行（性能约束、API 限制），回 P2 调整设计或回 P1 调整需求。

**产出：** 实现计划文档（bite-sized 步骤 + 精确文件路径 + 完整代码）

## Phase 4: 编码实现

**Goal：按计划执行，测试驱动，通过两阶段 Review。**

### Scope Guard（范围控制）

P4 开始前，用 TodoWrite 锁定 task 列表（来自 P3 计划）。执行过程中：
- 发现的新需求 → 标记为"后续迭代"，不当场做
- 发现的前置依赖 → 允许加入当前 task，但必须记录原因
- 范围膨胀信号：task 数量增加 > 50% → 暂停，汇报用户确认是否继续
- **调试时用 `/freeze` 锁定编辑范围**，防止修 bug 时误改其他模块
- **操作生产环境时用 `/careful`**，破坏性命令（rm -rf, force-push, DROP TABLE）自动拦截警告

### 并行分发（5+ 独立 task 时自动启用）

当 P3 计划包含 5+ 个**无共享状态**的独立 task 时，用 subagent 并行执行：

**判断标准：**
- ✅ 可并行：不同文件、不同模块、无数据依赖
- ❌ 不可并行：共享 state、文件互相 import、有执行顺序要求

**并行流程：**
1. 识别独立域（按模块/文件分组）
2. 每个 subagent 给精确的 task 描述 + 上下文 + 约束 + 期望输出
3. 并行分发（Agent tool, run_in_background）
4. 收集结果 → 验证无冲突 → 跑完整测试套件

**不可并行时串行执行，不要硬并行。**

### 4a. 编码（两条路径）

#### 路径 A：新功能 / 有测试的项目 → TDD

<HARD-GATE>
铁律 2：测试先行（中/大改动）。
没有失败的测试就不写生产代码。小改动（文案/颜色/间距）豁免。
- "太简单不需要测试" → 查改动规模表，是小改动才能跳过
- "我先写完再补测试" → 先写的测试回答"应该做什么"，后写的测试只能回答"做了什么"
- "已经手动测过了" → 手动测试 ≠ 系统测试
</HARD-GATE>

**Red-Green-Refactor 循环：**

1. **RED** — 写一个最小的失败测试，描述期望行为
   - 跑测试 → 必须 FAIL（且是正确的失败，不是语法错误）
   - 如果直接 PASS → 说明在测已有功能，修改测试
   - 如果 test error（不是 fail）→ 修 error，重跑直到正确 fail
2. **GREEN** — 写最小代码让测试通过
   - 不加功能、不重构、不"改进"
   - 跑**相关模块**测试 → 必须 PASS（全量测试留给 P5）
   - 测试没过？修代码，不是修测试
3. **REFACTOR** — 在测试全绿的基础上重构
   - 消除重复、改善命名、提取 helper
   - 跑测试 → 仍然全 PASS

**循环直到该 task 完成，然后下一个 task。**

**好测试的标准：**

| 特征 | 好 | 坏 |
|------|---|---|
| **最小** | 测一件事。名字里有 "and"？拆开 | `test('validates email and domain and whitespace')` |
| **清晰** | 名字描述行为 | `test('test1')` |
| **真实** | 用真实代码，mock 只在不可避免时用 | 测的是 mock 行为而不是真实代码 |

**先写了代码再补测试？Delete it. 从头开始。**
- 不要保留当"参考"
- 不要"改编"已有代码
- 不要看着它写测试
- Delete means delete

**卡住时怎么办：**

| 问题 | 解法 |
|------|------|
| 不知道怎么测 | 先写期望的 API 调用方式，再写断言 |
| 测试太复杂 | 说明设计太复杂，简化接口 |
| 必须 mock 所有东西 | 代码耦合太紧，用依赖注入 |
| 测试 setup 太大 | 提取 helper。还复杂？简化设计 |

**TDD 强制输出模板（每个 Red-Green 循环都要贴）：**

```
### RED
测试: <测试名称和期望行为>
运行: <测试命令>
结果: FAIL ✅ — <失败原因（必须是功能缺失，不是语法错误）>

### GREEN
改动: <写了什么代码>
运行: <测试命令>
结果: PASS ✅ — <全部测试通过数>
```

缺任何一段 = 没走 TDD，不能进 P5。
"测试先跑了 FAIL"没有命令输出证据 = 没有发生。

**TDD 完成检查清单：**
- [ ] 每个新函数/方法都有测试
- [ ] 每个测试都看到过 FAIL（有 RED 输出证据）
- [ ] 失败原因正确（功能缺失，不是 typo）
- [ ] 写了最小代码让测试通过（有 GREEN 输出证据）
- [ ] 所有测试全绿
- [ ] 测试用真实代码（mock 仅在不可避免时）
- [ ] 边界情况和错误场景已覆盖

#### 路径 B：Bug 修复 → 系统调试

<HARD-GATE>
铁律 3：根因先行。
找到根因之前，禁止提出任何修复方案。禁止"试试改这个看看"。

铁律 3b：锚定用户场景。
复现和验证必须匹配用户报告的操作。不允许用不相关的测试替代。
拿不到测试条件（如 auth token）不等于可以跳过，必须想办法拿到。
</HARD-GATE>

**Bug 修复强制输出模板（缺任何一段不能进 P5）：**

```
### 用户报告
<原文复述用户描述的操作和错误>

### 复现
<用匹配用户操作的命令/步骤复现，贴出看到的错误>
（如果无法复现 → 问用户要更多信息，不要猜测修复）

### 根因
<一句话说明根因>

### 修复
<代码改动说明>

### 验证
<用同样的操作再跑一次，贴出错误消失的证据>
```

**四阶段调试法：**

**阶段 1：复现用户场景**
1. 仔细阅读用户描述，提取精确操作路径（哪个页面、什么按钮、什么输入）
2. 用**完全匹配**的操作复现（用户说 POST 就测 POST，说历史页面就从历史页面进）
3. 拿不到前置条件（auth token、测试数据）→ **想办法拿到**，不要换一个更简单的测试
4. 看到同样的错误 → 阶段 2。看不到 → 问用户要更多信息

**阶段 2：查日志（先看线上证据，再猜本地原因）**
1. **Sentry** — 有没有对应时间段的新 error？stack trace 直接定位
2. **Vercel Function Logs** — `vercel logs <url>` 或 Vercel Dashboard → 函数日志
3. **浏览器 Console** — 用户端 JS 错误（让用户截图，或 Playwright 抓取）
4. **Supabase Logs** — Dashboard → Logs → API/Auth/Postgres，检查 4xx/5xx
5. 日志有明确错误 → 直接跳到阶段 3。没有 → 继续阶段 2b

**阶段 2b：代码层排查**
1. 检查最近改动（`git diff`、新依赖、配置变更）
2. 多组件系统：在每个边界打日志（什么进来、什么出去）
3. 追踪数据流（坏值从哪里来？一直往上追）
4. 找到正常工作的类似代码，逐行对比差异

**生产独有 bug（本地复现不了）：**
- 检查 env vars 差异（本地 `.env.local` vs Vercel production）
- 检查 Edge Function vs Node.js runtime 差异
- 加临时 console.error 到可疑位置 → 部署 → 看 Vercel logs → 删掉临时日志

**阶段 3：假设与验证**
1. 形成单一假设（写下来："我认为 X 是因为 Y"）
2. 最小测试（一次只改一个变量）
3. 验证：修好了？→ 阶段 4。没修好？→ 新假设

**阶段 4：实现修复 + 锚定验证**
1. 创建失败测试用例（最简复现）
2. 实现单一修复（只改根因，一次一个改动）
3. **用阶段 1 同样的操作验证** — 不是跑单元测试，是重走用户的操作路径
4. 贴出验证证据（命令 + 输出）

**如果修了 3 次还没好 → STOP。** 这可能是架构问题，不是假设失败。暂停并汇报用户。

**可选升级：** 复杂 bug 可调用 `/investigate`（gstack），自动启用 scope lock（freeze）防止调试时误改其他模块，并输出结构化 DEBUG REPORT。

**调试反模式（禁止）：**
- ❌ "先快速修一下，回头再查原因"
- ❌ "试试改 X 看看"
- ❌ "一次改多个东西，跑测试"
- ❌ "跳过测试，手动验证"
- ❌ "应该是 X 的问题，让我修一下"
- ❌ 修完一个 bug 立刻 deploy，拿 Vercel preview 当调试环境（每次 deploy 浪费 15-30s，n 次就是 n×30s）
- ❌ 发现第一个 bug 就立刻开修，忽略其余 bug（线性追 bug = 修了一个引出下一个，无限循环）

**调试正确流程补充：**
1. **先全面收集 bug，再下手** — 跑完所有测试，把所有失败/异常记录下来，通观全局，一次性制定修复计划
2. **本地验证优先** — 任何修复先在本地 `npm run dev` + curl/Playwright 验证通过，再 push。严禁用 Vercel preview 当 REPL
3. **测试脚本本身先验证** — E2E/集成测试脚本有 bug 时会掩盖 app bug。写完测试脚本，先用一个已知正确的 case 验证脚本行为符合预期，再大规模跑

### 4b. Code Review（自动执行，中/大改动）

P4 编码完成后，调用 `code-reviewer` skill 做一次审查：

- 有 P1/P3 的任务：同时检查 spec 合规 + 代码质量
- Bug fix（无 P1/P3）：只检查代码质量
- 小改动：跳过 review

按 🔴🟡💭 分级：
- **🔴 Blocker** → 必须修复，回 4a。不修不能进 P5。
- **🟡 Suggestion** → 应该修复，当场改。
- **💭 Nit** → 可选，不阻塞。

**Review 原则：**
- 不要表演式同意，有理有据地修复或反驳
- YAGNI 检查：没人调用的就不实现

**↑ 挑战 P3：** 编码过程中如果发现架构设计过度或依赖不可用，回 P3 简化。

## Phase 5: 验证检查

**Goal：发布前全面验证，用证据说话。**

<HARD-GATE>
铁律 4：验证先行。
没有跑验证命令就不能声称"通过了"/"完成了"/"没问题"。

验证 5 步法：
1. IDENTIFY — 什么命令能证明这个声明？
2. RUN — 执行完整命令（新鲜的、完整的）
3. READ — 读完整输出，检查 exit code，数失败数
4. VERIFY — 输出是否支持声明？
   - 否 → 如实报告实际状态 + 证据
   - 是 → 带证据声明
5. CLAIM — 只有到这一步才能说"通过了"

跳过任何一步 = 在撒谎，不是在验证。
</HARD-GATE>

**禁止用语（验证前）：**
- ❌ "应该没问题" / "大概通过了" / "看起来可以"
- ❌ 跑完就说 "Great!" / "Perfect!" / "Done!"（先看输出）
- ❌ 信任之前的运行结果（必须重新跑）
- ❌ 信任 subagent 的成功报告（自己验证）

**验证清单：**

| 声明 | 需要的证据 | 不充分的证据 |
|------|-----------|-------------|
| 测试通过 | 测试命令输出：0 failures | 之前的运行、"应该过" |
| Lint 干净 | Lint 输出：0 errors | 部分检查、推测 |
| 构建成功 | Build 命令：exit 0 | Lint 通过就算构建通过 |
| Bug 已修 | P4 模板中的验证证据已贴出 | 改了代码就算修了 |

**按需执行的检查项：**

1. **安全检查** — `git diff --cached` 中无 `.env*`、API key、token
2. **设计审查** — 有 UI 改动时执行
3. **无障碍审计** — 有 UI 改动时执行
4. **性能检查** — 新功能或大改动时执行
5. **Metadata 检查** — 有页面变动时执行

**调用 skill：**
- `security-review` → 安全审计
- `design-audit` → UI/UX 设计审查
- `accessibility-auditor` → WCAG 2.2 AA 审计
- `performance-benchmarker` → Core Web Vitals + Bundle
- `fixing-accessibility` → 快速无障碍修复
- `fixing-metadata` → meta 标签完整性
- `fixing-motion-performance` → 动画性能

**按需执行：** 不是每个 skill 每次都调，根据改动范围自动判断。

**↑ 挑战 P4 和 P2：**
- 安全问题 / 代码质量问题 → 回 P4 修复代码
- 设计问题（布局、颜色、间距不一致）→ 回 P2 调整设计
- 无障碍问题 → 当场用 `fixing-accessibility` 修复（不回弹）

**产出：** 所有检查通过（带验证证据）

## Phase 6: 发布上线

**Goal：代码上线生产环境。**

### 路径 A：有 Playwright（完整流程）

```
main ─────────────────────────── merge ──→ production
  \                                /
   └─ dev ─→ push ─→ Preview ─→ e2e ─┘
                       ↑              |
                       └── 失败修复 ←─┘ (最多 3 次)
```

**调用 skill：** `release-flow`

**↑ 挑战 P4：** e2e 测试失败 → 分析失败原因（走 4a 调试路径）→ 修复 → 重新 push → 重新测试（最多 3 次）

### 路径 B：无 Playwright（简化流程）

```bash
git add <files> && git commit -m "<type>: <desc>" && git push origin main
```

Vercel 自动部署 production。

### 异步部署等待（Fire-and-Forget）

Push 后 Vercel 部署通常需要 30-90 秒。**不要干等**，利用这段时间并行工作：

```
git push origin main
  ↓
  ├─ [后台] 等待 Vercel 部署完成（run_in_background）
  └─ [前台] 同时执行：
       - 更新 PIPELINE_STATE.json（phase: "P7"）
       - 检查文档是否需要更新（P7.4）
       - 准备 smoke test 命令
       - 检查 Sentry 基线（P7.2）
  ↓
  部署完成通知到达 → 执行 P7 上线验证
```

**规则：** 后台任务完成后再执行需要线上环境的验证步骤（curl、browse、e2e）。

### 回滚策略

**每次部署前记录回滚点，出问题秒级恢复。**

**部署前：**
```bash
# 记录当前生产版本的 commit hash
ROLLBACK_SHA=$(git rev-parse HEAD~1)
```

**P7 发现问题时的回滚流程：**
```bash
git revert HEAD --no-edit    # 创建 revert commit（不破坏历史）
git push origin main         # 触发 Vercel 重新部署到上一版本
```

**回滚后：** 回 P4 修复 → 重新走 P5→P6→P7，不要在回滚的基础上 patch。

**规则：**
- 自动进入，不问用户
- E2E 失败自动修复重试（最多 3 次）
- 不手动 `vercel --prod`，push 即可
- 每个子目录是独立 git repo
- P7 发现生产问题 → 先回滚再修复，不要让用户等修复

## Phase 7: 上线验证

**Goal：用证据确认生产环境正常。**

<HARD-GATE>
铁律 4 仍然适用。不跑验证就不能说"部署成功了"。
</HARD-GATE>

### 7.1 部署确认

1. `vercel ls` 确认 production 部署成功 → 贴输出
2. 确认 production URL 可访问 → 贴状态码
3. **（推荐）** 用 `/browse` 打开线上 URL，截图确认页面渲染正常、核心交互可用（比 curl 200 更可靠）

### 7.2 场景回归（Bug 修复时必做）

<HARD-GATE>
如果本次部署是 bug 修复，必须在生产环境重走 P4 模板中"用户报告"的操作路径。
不是 health check，不是 curl 200，是**用户当时失败的那个操作**。
验证通过 → 贴证据。验证失败 → 回 P4，不要告诉用户"已修复"。
</HARD-GATE>

### 7.3 运行时观测（有 Sentry 的项目）

部署后检查运行时健康状态：
- **Sentry 错误** — 部署后是否有新增 error（对比部署前后）
- **关键 API smoke test** — 有 API 的项目，用 curl 打一个健康检查请求
- 发现异常 → 立即回 P4 修复，不要等用户报告

**项目 Sentry 配置：**

| 项目 | 有 Sentry | 有 API |
|------|----------|--------|
| plat | ✅ | ✅ (parse, generate-image) |
| museum | ✅ | ❌ |
| portfolio | ✅ | ❌ |
| 其他 | ❌ | 按项目判断 |

### 7.4 成功指标回顾

对照 P1 设计文档中的 **Success Criteria**，逐条输出：

```
### 成功指标验证
| # | 标准 | 达成? | 证据 |
|---|------|-------|------|
| 1 | <P1 定的标准> | ✅/❌ | <具体命令输出/截图/数据> |
| 2 | ... | ... | ... |
```

- 每条必须有对应证据，不能笼统说"达标了"
- ❌ 的标记为后续迭代项，不阻塞本次发布
- 没有 P1 成功标准的（小改动/bug fix）→ 跳过本节

### 7.5 活文档同步

**代码改了，文档必须跟着改。** 检查以下文件是否需要更新：

| 文件 | 检查条件 | 更新内容 |
|------|---------|---------|
| `README.md` | 新项目 / 新功能 / 新依赖 / 新环境变量 | 项目说明、使用方式、第三方服务依赖 |
| `CLAUDE.md`（根目录） | 新项目 / 项目能力矩阵变化 | 项目表、命令、环境变量、服务依赖 |
| 设计文档 | 实现偏离了设计 | 标注实际实现与设计的差异 |

**规则：**
- 新项目必须有 README（至少包含：是什么、怎么跑、环境变量、第三方服务依赖）
- 不要写过度的文档，够用即可
- 文档更新和代码一起 commit，不要分开

### 7.6 汇报用户

改动内容 + 验证证据 + production URL + 运行时状态 + 成功指标达成情况

## 自执行检查点系统

**每个阶段结束前，Pipeline 自动执行检查点验证。不通过就回退，不需要用户提醒。**

### 检查点机制

每完成一个阶段，自动执行以下自检：

```
阶段完成 → 自检清单 → 全过？→ 下一阶段
                         ↓ 不过
                    自动修复 → 重新自检（最多 2 次）
                                  ↓ 仍不过
                             汇报用户
```

### 各阶段检查点

| 阶段 | 自检项 | 验证命令 |
|------|--------|---------|
| P4 编码 | TypeScript 编译 | `npx tsc --noEmit` |
| P4 编码 | 测试全绿 | `npm run test` |
| P4 编码 | Lint 无 error | `npm run lint` (如有) |
| P5 验证 | 构建成功 | `npm run build` |
| P5 验证 | 无密钥泄露 | `git diff --cached \| grep -iE 'sk-\|AIza\|api_key'` |
| P6 发布 | Push 成功 | `git push` exit code |
| P6 发布 | 部署完成 | `vercel ls` 状态 Ready |
| P7 验证 | 线上可访问 | `curl -sI <url>` 返回 200 |
| P7 验证 | API 正常 | smoke test (如有 API) |

### 自检失败自动修复

1. **TypeScript 编译失败** → 读错误 → 修复类型 → 重新编译
2. **测试失败** → 调用 `auto-fix` skill 自主修复循环
3. **Lint 失败** → `npm run lint -- --fix` 自动修复
4. **构建失败** → 读错误日志 → 修复 → 重新构建
5. **部署失败** → 查 `vercel logs` → 修复 → 重新 push
6. **线上异常** → 回滚 (`git revert HEAD`) → 回 P4 修复

### 违规检测（自我监督）

Pipeline 运行中持续检测以下违规行为，发现即停：

| 违规 | 检测方式 | 处理 |
|------|---------|------|
| 跳过测试 | 进入 P5 但没有 test 命令输出 | 回 P4 补测试 |
| 未验证就声明完成 | 说"完成"但没有命令输出证据 | 撤回声明，跑验证 |
| Push 后没跑 e2e | 有 playwright 但跳过 release-flow | 立即执行 release-flow |
| 用了旧依赖 | package.json 中无该依赖但 import 了 | 停下来确认技术栈 |

## 全局规则

1. **全自动执行** — 不要问用户"要不要执行 Phase X"（但 bug fix 复现信息不足时必须问用户，这不算违反自动执行）
2. **智能跳过** — 没有 UI 改动不跑设计审查，没有 lint 不跑 lint
3. **失败即修** — 任何阶段失败都先自动修复，不要停下来问
4. **挑战不过就打回** — 下游发现上游问题，直接回上游修，不要带病前进
5. **最多回弹 2 次** — 超过 2 次说明方案有根本问题，暂停汇报用户
6. **安全第一** — commit 前必查密钥泄露
7. **闭环进化** — P7 完成后自动触发 P8（evolve），形成完整闭环
8. **一次一问** — P1 阶段每条消息只问用户一个问题
9. **证据说话** — 任何"完成"/"通过"声明必须附带验证命令输出
10. **自执行检查点** — 每阶段结束自动验证，不通过自动修复，不需要用户提醒
11. **违规自检** — 持续检测跳步/未验证/旧依赖等违规行为，发现即停
