# TDD Reference

在 `P4a` 进入“新功能 / 中大改动 / 重构 / 有测试项目”路径时读取本文件。

## 目标

先写失败测试，再写最小实现，再重构。没有失败测试，不写生产代码。

## Red-Green-Refactor

1. **RED**
   - 写一个最小失败测试，描述期望行为
   - 运行测试，必须 `FAIL`
   - 失败原因必须是“功能缺失”，不是语法错误
2. **GREEN**
   - 写最小代码让测试通过
   - 只修当前失败，不顺手加功能
   - 运行相关模块测试，必须 `PASS`
3. **REFACTOR**
   - 在测试全绿的前提下消除重复、改善命名、提取 helper
   - 重跑测试，仍然 `PASS`

## 好测试标准

- 一次只测一件事
- 名字描述行为，不写 `test1`
- 优先真实代码，mock 只在不可避免时使用
- 边界条件和错误场景有覆盖
- UI / E2E 测试必须包含“操作 → 结果变化”，不能只检查元素存在

## 反模式

- “先写代码再补测试”
- “太简单不需要测试”但实际不是小改动
- “已经手动测过了”
- 测试不失败就直接进入实现
- UI / E2E 里只断言元素存在，不断言点击后的页面、状态或数据变化

## 卡住时怎么处理

| 问题 | 处理 |
|------|------|
| 不知道怎么测 | 先写期望 API / 使用方式，再写断言 |
| 测试太复杂 | 说明设计太复杂，先简化接口 |
| 必须大量 mock | 说明耦合过高，考虑依赖注入 |
| setup 太大 | 提取 helper；还太大就回 P3 简化 |

## 强制输出模板

```text
### RED
测试: <测试名称和期望行为>
运行: <测试命令>
结果: FAIL ✅ — <失败原因，必须是功能缺失>

### GREEN
改动: <写了什么代码>
运行: <测试命令>
结果: PASS ✅ — <通过数量>
```

缺任何一段，就不算完成 TDD。

## 完成检查清单

- 每个新增函数/行为都有测试
- 每个测试都看过一次真实 `FAIL`
- `FAIL` 原因正确
- 有最小实现让测试转绿
- 所有相关测试最终全绿
