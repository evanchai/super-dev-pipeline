# Debug Reference

在 `P4a` 进入“bug 修复 / 根因不明 / 线上问题”路径时读取本文件。

## 核心原则

- 先复现，再定位根因，再修复
- 不允许“试试改这个看看”
- 复现与验证必须匹配用户原始操作路径

## 强制输出模板

```text
### 用户报告
<原文复述用户操作与报错>

### 复现
<按用户路径复现，贴出错误>

### 根因
<一句话说明根因>

### 修复
<代码改动说明>

### 验证
<用同一路径再跑一次，贴出错误消失的证据>
```

## 四阶段调试法

### 1. 复现用户场景

- 提取精确路径：页面、按钮、输入、前置条件
- 完全按用户路径复现
- 拿不到 token / 测试数据，不等于可以跳过；先想办法拿到
- 复现不了就向用户补信息，不要猜修复

### 2. 查日志

- Sentry / 平台日志 / 浏览器 console / 数据库日志
- 日志有明显错误就直接收敛到假设
- 本地复现不了时，重点看环境差异和运行时差异

### 3. 单一假设验证

- 一次只验证一个假设
- 一次只改一个变量
- 两次假设失败后，自动升级 `/investigate`

### 4. 实现修复并锚定验证

- 写最简失败测试用例（如适用）
- 一次只修一个根因
- 用阶段 1 的同一路径重跑验证

## 自动升级条件

满足任一条件，自动调用 `/investigate`：
- 根因不明
- 线上独有
- auth / payment / data 问题
- 连续两次假设失败

## 调试反模式

- 先修后查
- 一次改多个点
- 用 preview 当 REPL
- 只跑 smoke，不重走用户路径
- 发现第一个 bug 就立刻修，忽略全局失败面

## STOP 条件

同一 bug 连续修 3 次还没好，暂停并汇报用户。此时更像架构问题，不是假设没试够。
